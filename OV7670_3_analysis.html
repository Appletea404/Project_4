종합 판정: 동작 가능하나 수정 권장

전체 파이프라인 구조(카메라 → SCCB 초기화 → BRAM → VGA 출력)는 완성되어 있습니다. RGB 변환, 이중 버퍼링, VGA 타이밍도 의도대로 구현되어 있습니다. 다만 modport 방향 오류(Vsync), 다중 드라이버(wt_valid), SCCB 초기화 실패 시 무음 중단 등 3가지 중요 이슈가 있어 특정 조건에서 오동작할 수 있습니다.

이중 버퍼링 구조

BRAM 144,000픽셀 = 0번 방(0~71999) + 1번 방(72000~143999). room_chk 신호로 쓰기/읽기 방을 교대 전환합니다.

room_chk = 0 일 때

쓰기(Image_conv) → addra = 72000~143999 (1번 방)
읽기(VGA_ctrl) → addrb = 0~71999 (0번 방)

room_chk = 1 일 때

쓰기(Image_conv) → addra = 0~71999 (0번 방)
읽기(VGA_ctrl) → addrb = 72000~143999 (1번 방)

Vsync rising edge 감지 시 room_chk 토글. finish 신호로 VGA 컨트롤러에 프레임 완료 통보. room_chk는 PCLK 도메인에서 생성되므로 VGA_controller_b에서 2단 FF로 동기화 처리됨.

발견된 이슈 목록

중요도: 중 Bug #1 — Vsync modport 방향 오류
VGA.sv의 bot modport에서 Vsync가 input으로 선언되어 있으나, VGA_controller_b 내부에서 mi0.Vsync <= ...로 직접 드라이브합니다.
방향이 반전되어 Vivado에서 modport 위반 경고가 발생합니다. synthesis는 통과할 수 있으나 의도와 다른 선언입니다.
수정 방법: bot modport에서 Vsync를 output으로 변경.
위치: VGA.sv:95 (modport bot), VGA_controller.sv:87
중요도: 중 Bug #2 — wt_valid 다중 드라이버 충돌
인터페이스 내부에 assign wt_valid = 1'b1; 상수 드라이브가 존재하면서, 동시에 image_cov modport는 wt_valid를 output으로 선언하여 Image_conv가 이를 procedural 제어 시도합니다.
결과적으로 wt_valid는 항상 1로 고정되어 BRAM의 wea가 매 클럭 활성화됩니다. address가 고정된 구간에서는 동일 데이터를 반복 쓰므로 기능적 파괴는 없으나, 불필요한 전력 소모 및 잠재적 타이밍 이슈가 있습니다.
수정 방법: 인터페이스의 assign wt_valid = 1'b1; 제거 후 Image_conv 단독 제어로 변경.
위치: VGA.sv:54 (인터페이스 내 assign), Image_conv.sv:50,84,95,105
중요도: 중 Bug #3 — SCCB 초기화 실패 시 무음 중단
OV_MASTER의 COM1 상태에서 레지스터 Write 후 Read-back 값이 불일치하면 loop_en <= 0만 설정하고 종료합니다. 이 경우 sccb_master_done이 영원히 0을 유지하여 Image_conv가 초기화 완료를 기다리며 무한 대기 상태에 빠집니다.
LED로 구분이 안 되어 하드웨어에서 디버깅이 어렵습니다.
수정 방법: 실패 시 재시도 로직 추가 또는 별도 실패 LED 표시 추가 권장.
위치: OV_MASTER.sv:139~141 (COM1 상태 else 분기)
정보 Bug #4 — PCLK 전용 클럭 네트 미사용
XDC에 set_property CLOCK_DEDICATED_ROUTE FALSE [get_nets PCLK_IBUF]가 설정되어 카메라 PCLK 신호가 전용 클럭 배선(Global Clock Network) 대신 일반 배선을 통해 분배됩니다.
Basys3의 JC1(K17)이 전용 클럭 입력 핀이 아니기 때문에 불가피한 선택이나, 타이밍 마진이 감소하고 clock skew가 커질 수 있습니다.
필요 시 전용 BUFG/BUFR를 명시적으로 삽입하거나, 클럭 전용 핀(e.g. W5)이 있는 커넥터로 이동을 검토하세요.
위치: Basys-3-Master.xdc:99
경고 (기능 정상) Bug #5 — RGB 변환 비트 폭 불일치 (Vivado 경고)
Image_conv.sv에서 두 가지 폭 불일치가 존재합니다.
① {red[4:0], green[2:0]} <= {image_data[7:4], image_data[2:0]} : LHS 8bit, RHS 7bit (0 패딩)
② in_dt[11:0] <= {red, green, image_data[7], image_data[4:1]} : LHS 12bit, RHS 13bit (MSB 절삭)
Vivado는 경고를 출력하지만, 절삭/패딩 후의 결과는 R[3:0]/G[3:0]/B[3:0]의 RGB444 포맷과 일치합니다. 기능적으로는 정상 동작합니다.
위치: Image_conv.sv:92, 96
경고 Bug #6 — led_data 미선언 포트 참조
OV_VGA 최상위 모듈에서 assign led_data = main_bus.led_data;를 사용하지만, led_data가 포트 목록에 없습니다. Vivado에서 implicit wire 경고가 발생하며 실제 출력 핀에 연결되지 않아 외부에서 관찰 불가능합니다.
수정 방법: 불필요하면 해당 assign 줄 삭제.
위치: VGA.sv:144

RGB565 → RGB444 변환 상세

OV7670은 RGB565를 2바이트로 전송합니다. Image_conv의 변환 로직을 정리하면 다음과 같습니다.

바이트Camera D[7:0]비트 의미처리
1번째 (rgb=0) image_data[7:3] = R[4:0]
image_data[2:0] = G[5:3]		 RGB565 상위 바이트 red <= {0, image_data[7:4]}
green <= image_data[2:0]
2번째 (rgb=1) image_data[7:5] = G[2:0]
image_data[4:0] = B[4:0]		 RGB565 하위 바이트 in_dt[11:0] = {R[3:0], G[5:3], G[2], B[4:1]}
→ RGB444 완성

최종 BRAM 저장 포맷: in_dt[11:8] = R[3:0]  |  in_dt[7:4] = G[3:0]  |  in_dt[3:0] = B[3:0]
VGA 출력: vgaRed = doutb[11:8], vgaGreen = doutb[7:4], vgaBlue = doutb[3:0]
